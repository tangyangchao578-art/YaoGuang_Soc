# 将创意头脑风暴转化为设计

## 概述

通过自然的协作对话将创意转化为成熟的设计和规格。

首先了解当前项目上下文，然后逐一提问以完善创意。一旦理解了要构建的内容，分小节（200-300字）展示设计，每节后确认是否正确。

## 流程

**理解创意：**
- 首先查看当前项目状态（文件、文档、最近提交）
- 逐一提问以完善创意
- 尽可能使用选择题，但开放式问题也可以
- 每条消息只问一个问题 - 如果主题需要更多探索，分解为多个问题
- 关注理解：目的、约束、成功标准

**探索方法：**
- 提出2-3种不同方法及其权衡
- 以对话方式展示选项，并附上建议和推理
- 以推荐选项为先，解释原因

**展示设计：**
- 一旦理解了要构建的内容，展示设计
- 分解为200-300字的章节
- 每节后询问目前是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 如果某处不合理，准备回溯澄清

## 设计之后

**文档编写：**
- 将验证的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如有可用，使用 elements-of-style:writing-clearly-and-concisely 技能
- 将设计文档提交到 git

**实现（如果继续）：**
- 询问："准备好设置实现了吗？"
- 使用 superpowers:using-git-worktrees 创建隔离工作空间
- 使用 superpowers:writing-plans 创建详细实现计划

## 关键原则

- **一次一个问题** - 不要用多个问题压倒用户
- **优先选择题** - 尽可能用选择题，比开放式问题更容易回答
- **无情的YAGNI** - 从所有设计中移除不必要的功能
- **探索替代方案** - 在确定前始终提出2-3种方法
- **增量验证** - 分节展示设计，逐节验证
- **灵活调整** - 遇到不合理时回溯澄清
